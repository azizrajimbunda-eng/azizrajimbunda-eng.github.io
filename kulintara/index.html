One set of books,
done right.

Kulintara is a unified accounting system modeled on how Philippine government agencies actually keep their books — the Government Accounting Manual, the UACS chart of accounts, and the COA document flows, from official receipt to trial balance.

The honest fine print: Kulintara is an independent demonstration prototype. Every name and figure in the screenshots is fictitious seed data.

The concept

Government accounting in the Philippines is not freeform bookkeeping — it is a tightly specified discipline. The Government Accounting Manual (GAM) says how the books are kept, the Unified Accounts Code Structure (UACS) says what every account is called, and money moves through a fixed cast of documents: obligation requests, disbursement vouchers, official receipts, deposits, journal entry vouchers.

In many offices those flows still live across separate tools — a spreadsheet here, a logbook there, a standalone app for one function that doesn't talk to the next. Kulintara asks a simple question: what would it look like if the whole framework lived in one system — one place where a receipt, a voucher, and the trial balance are the same data, viewed from different angles?

The design goal wasn't "an app with accounting features." It was to model the rulebook faithfully: gapless document numbering, an append-only ledger, corrections by reversal instead of edits, roles that mirror how a real finance office divides its work — accountant, cashier, budget officer, admin, viewer.

How it was built

  1. Domain model first, screens later

    Before any interface existed, the document lifecycles were written down as rules: a DV moves draft → for approval → approved → paid, and each transition is enforced server-side. An ORS freezes its fund cluster and date the moment a number is issued, because the number encodes both.

  2. Money as exact decimals, never floats

    Every amount is a decimal handled with precise arithmetic all the way through — even aggregates coming back from SQL pass through a single decimal helper. At scale, a cent lost to rounding is a cent an auditor will ask about.

  3. Document numbers that can survive a race

    Gapless numbering is only trustworthy if two people saving at the same instant can't get the same number. Allocation happens per fund cluster and period under a database lock, verified with 200 concurrent allocations: 200 unique, contiguous numbers. In a gapless series, a missing number is evidence; a duplicate is a defect.

  4. Tests on the engine production actually uses

    226 tests cover the accounting behaviour, and they run on PostgreSQL — not a lightweight stand-in. CI runs the suite plus linters on every push.

  5. Screenshots as a build artifact

    Even the documentation screenshots are generated by a script against freshly seeded demo data, so they can be regenerated after any interface change instead of quietly going stale.

What fought back

The date bug hiding in the test database

The test suite originally ran on SQLite because it's fast. But SQLite has no real DATE type — a date column round-trips as a midnight timestamp string, and a date-range filter compared against it silently drops the last day of the period. For an accounting system, that is not a rounding error; that is a wrong trial balance. The suite moved to PostgreSQL, the engine production uses.

The safety setting that didn't work

Moving tests to a persistent database created a new danger: Laravel's test refresh drops every table in whatever database is configured. Misconfigure the connection and the suite isn't just wrong — it's destructive. PHPUnit's documented answer, force="true" on the env config, looked right, so it went in with a confident comment. Then it got tested: point the suite at a database that shouldn't exist, run it. It happily created and migrated that database. The "fix" wrote to two places Laravel reads after the one the shell already owns. The real fix asks the resolved connection itself for its database and driver, and refuses to run against anything unexpected. A safety mechanism you have not seen fail is just an assumption with good branding.

Immutability vs. convenience

The tempting design is to let users edit posted entries — it feels helpful. The correct design is the uncomfortable one: once posted, a journal entry can never be edited or deleted. Corrections happen by reversal, with a required reason, linked back to the original. Both stay in the ledger forever. An auditor can always see what was recorded, what corrected it, and why.

Free and open source

Kulintara is free software under the AGPL-3.0 — run it, study it, modify it, redistribute it. The full source, design docs, user guide, and engineering notes are public. Contributions are welcome; the repo has a contributing guide and a code of conduct. Deploying it for real government books would require the proper COA evaluation and approval — that responsibility sits with whoever deploys it.
